# Database Migration Order

**Owner:** Atlas · **Phase:** 5 · **Gate:** every migration tagged with its milestone (PRD §16) and applied forward-only per `agents/data/atlas.md` rule 1 ("Schema changes are the most expensive changes. Take time to get them right.").

Migrations are numbered with a stable 4-digit prefix. **Never rewrite a shipped migration** — superseding changes ship as new files. Per-file rationale below; each milestone's migrations are gated by the corresponding PRD §16 exit gate.

## Why this order

**M0 (0001) — schema scaffold.** Atlas's rule 5: "Test migrations against production-like data, not an empty database." The initial migration is intentionally _complete shape-wise_ so M1 can ship RLS against the real tables instead of evolving schema alongside policies. This avoids the motionmax pattern of "policies that assumed a column that didn't ship yet." The seed row for `score_engine_versions` lets Verify's contract test run against a DB-backed engine on day one.

**M1 (0002) — RLS lockdown + runner JWT minting.** RLS is enabled _after_ the schema is stable so policy-design reviewers can read the policy file against a fixed table file. This also lets us prove the _negative_ case in Verify's `integration/rls-cross-tenant.spec.ts` (PRD §18.5 Goal 5): Tenant A's JWT cannot SELECT Tenant B's findings row. The mint endpoint ships in the same migration as the policies so the `auth.tenant_id()` helper has a real producer end-to-end — see `runner-jwt.md`.

**M2 (0003) — Managed billing tables + Stripe idempotency.** Decision D6 (locked v0.4) reorders Managed before CLI. The `billing_events.stripe_event_id UNIQUE` constraint (shipped in 0001) is the idempotency primitive (closes Ledger's billing-and-cancel.md Unhappy 4 acceptance criterion). M2 Batch 1 (landed) adds the columns Forge's Stripe webhook persists to (`stripe_customer_id`, `plan_family`, `billing_period`), the D22 reset counter on `cooling_off_windows`, the R1 cap-check (`tenant_token_usage_daily` + `check_token_budget`), the AH-6 `projects.client_tag`, and the carry-over `enqueue_audit_run` RPC that route.ts has been waiting on since M1. D20's auto-open-cooling-off trigger remains on the Batch 2 backlog (it lives in trigger logic, not table shape — so it can ship in a successor 0003a or 0004 once Forge's webhook handler agrees on the open-event payload).

**M3 (0004) — CLI pairings.** Decision D6 reorders CLI to M3. The pairing-code-replay defence ships here, along with replay-attempt counters and revocation-undo plumbing for the 5-min undo window in `settings-and-account-management.md` S-CLI.

**M4 (0005) — Lifecycle email triggers.** E1–E5 sequences need a pg_cron-backed scheduler that fires on `runs.verdict = 'FAIL'` + `runs.product = 'surface'` (E2), `runs.verdict = 'PASS WITH FIXES'` (E3), etc. The scheduler is a Postgres job — putting it here rather than in app code is deliberate: email cadence is a data-layer SLO, and the database job survives app deploys. `audit_log_write()` becomes callable from the app layer at this point (it existed since 0002 but was service-role-only) — required by S-CLI revoke flow + the URL-audit attestation log.

**M5 (0006) — DMCA + retention triggers.** DMCA takedown lifecycle (intake → review → action → audit) ships at M5 because of the DMCA Designated Agent registration (PRD §14.5). `audit_logs` is partitioned at this point to keep the 7-year retention performant — partitioning earlier would have been premature; partitioning later would have meant a costly online repartition. 90-day pg_cron purge for `data_exports.expires_at` per `settings-and-account-management.md` EC-6.

**V1.5 (0008) — Auto-PR fix delivery.** Decision D3 graduates from specs-only to Auto-PR at V1.5. The schema upgrade is additive on `fix_pr_jobs` (0001 shipped a scaffold) and adds the load-bearing `fix_pr_jobs_never_default_branch` CHECK + the ARCH-D7 jury-gate-only RLS predicate. The Art. 50 disclosure machinery is application-side; the schema's job is to make the negative-case tests bulletproof and to give the Stripe + GitHub webhook handlers a place to land their refund + merge-state writes.

**V2 (0009) — Build mode.** PRD §4 V2 Goals + §7.3 Build Workflow. Build mode is a parallel aggregate root to `runs` — it shares the ULID-as-text PK pattern and the tenant/user scoping, but the lifecycle is its own 15-state machine (`build_state`) and the deliverable is a bundle (multi-artifact + seeded repo), not a verdict. The schema lands three new tables (`builds`, `build_artifacts`, `build_events`) and one new auth helper (`auth.runner_build_id()`). The load-bearing predicate is the analog of ARCH-D7: only the `build-audit-gate` Edge Function (JWT iss + role) may transition `builds.state` to `audit_gate_passed` or `audit_gate_failed`, paired with the `builds_delivered_verdict_required` CHECK that refuses delivery without a PASS / PASS WITH FIXES verdict. Together they make the V2 audit-gate-blocks-delivery negative test (`tests/integration/build-audit-gate.spec.ts`) bulletproof — the DB rejects an unsafe transition even if app code is buggy. `build_events` is append-only and is the substrate Vega's live build dashboard reconstructs from (V2 exit gate). Service-role bypass is the only INSERT path for artifacts + events (immutability); members SELECT only.

**V2.1 (0010) — Scaffold generation.** PRD §4 V2 Goal 8 + §7.3 step 5 third output preference (`roadmap_docs_seed_scaffold`) + `sprint/milestone-V2-1.md` exit gates. Scaffold/MVP code generation is gated behind audit-pass and ships as a child aggregate of `builds`: one `scaffold_jobs` row per build that elected the third output preference (FK CASCADE on the parent build + on the parent tenant). The lifecycle is a 10-state machine (`scaffold_state`) distinct from `build_state` because scaffold has its own intermediate detection + generation + packaging phases that the bundle assembler does not. The audit gate is the same Jury rubric but a separate run row (FK `scaffold_audit_run_id` → `runs.id`), so the scaffold can fail audit without poisoning the parent build's bundle delivery. The load-bearing predicate mirrors 0009 exactly: only the `scaffold-audit-gate` Edge Function may transition `scaffold_jobs.state` to `audit_passed` or `audit_failed`, paired with the `scaffold_jobs_delivered_verdict_required` CHECK (state=delivered ⇒ verdict IN PASS/PASS WITH FIXES + delivered_at + generated_artifact_id NOT NULL). `generated_artifact_id` references `build_artifacts(id)` — Studio Zero's canonical Build-mode storage-object registry — so scaffold artifacts live in the same bucket and inherit the same sha256 tamper-detection witness as bundle artifacts. The conceptual "storage_objects" table from the V2.1 spec is realized in Studio Zero's schema as `build_artifacts`. Build extension §3 (V2.1 scaffold IP amendment) makes the customer-IP-ownership posture identical to V2; the audit gate hard rule is documented in the extension as the load-bearing customer promise.

## Conventions

1. **Forward-only.** Every migration is wrapped in `BEGIN; … COMMIT;` and tested against a clone of the staging dataset before merging (`agents/data/atlas.md` rule 5). Never modify a shipped migration — supersede with a new file.
2. **One concern per migration.** A migration either changes schema OR adds policies OR adds data — never two at once. Reviews are reviewable.
3. **Destructive changes ship as multi-step.** Per Atlas's rule: (1) add new column → (2) backfill → (3) switch reads → (4) drop old column. Each step is its own migration file with its own milestone tag.
4. **Seed data is separate from schema.** `0001_initial.sql` adds the v1 `score_engine_versions` seed row at the _end_ of the transaction, after the table exists; further seed data ships in `0007_seed_demo_data.sql` (post-M5, for the demo tenancy used in marketing screenshots).
5. **Every migration declares its rollback plan.** Even when forward-only, the rollback is documented in the file header so the on-call has a path if a deploy lands a corrupted state. Rollback for `0001` = drop schema, restore from snapshot (this is what the staging-clone test exists to prevent).

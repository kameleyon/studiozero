## §1 — Brand Assets

### Blockers

**[1.B1] OG image is 2000×2000, declared as 1200×630.**
- File: `public/og-image.png` (2000×2000), `index.html:36-38`, `marketing/src/layouts/BaseLayout.astro:56-58`
- Evidence: `og:image:width=1200`, `og:image:height=630` meta tags (`index.html:37-38`); actual PNG dimensions 2000×2000 (per `file(1)` above).
- Impact: Twitter `summary_large_image`, Facebook, LinkedIn, Slack, iMessage, Discord, WhatsApp will either crop to a centered square (cutting off any landscape composition) or reject the asset for failing platform validators. Marketing share previews will look unprofessional on every channel from day one.
- Fix: produce a real 1200×630 OG image with MotionMax wordmark + tagline + aqua/gold gradient; replace `public/og-image.png` and `marketing/dist/og-image.png` (regenerated by build); keep current asset cache-busted via `?v=` bump.
- Effort: S

**[1.B2] Apple touch icon is 2000×2000 / 752 KB, declared 180×180.**
- File: `public/apple-touch-icon.png`, `index.html:25` (`<link rel="apple-touch-icon" sizes="180x180">`)
- Evidence: `file public/apple-touch-icon.png` → 2000×2000; `ls -la` → 752,294 bytes.
- Impact: every iOS user who pins to home-screen downloads 752 KB to render a 180×180 icon. On 4G, this adds ~2 s before the icon paints. Lighthouse PWA audit fails on icon-size mismatch. iPad and visionOS share the same path.
- Fix: generate true 180×180 (and 167×167 iPad, 152×152 iPad legacy, 120×120 iPhone) PNGs from the master logo with proper `apple-touch-icon-precomposed.png` fallback; reference each via separate `<link rel="apple-touch-icon" sizes="…">` in `index.html`.
- Effort: S

**[1.B3] Favicon is 2000×2000 / 752 KB.**
- File: `public/favicon.png`, `index.html:21-24` (`<link rel="icon" type="image/png" sizes="32x32" href="/favicon.png">`, `sizes="16x16"`, `<link rel="shortcut icon">`)
- Evidence: same SHA256 as above; 2000×2000.
- Impact: every HTML page request triggers a 752 KB favicon download to render a 16×16 tab icon. On a cold cache, the browser tab title shows the generic globe until the entire PNG arrives. There is no `favicon.ico` and no SVG favicon — modern browsers will not auto-fall-back.
- Fix: ship `public/favicon.ico` (multi-resolution 16/32/48), `public/favicon-16.png`, `public/favicon-32.png`, and `public/favicon.svg` (mask-icon-compatible). Replace the three `<link rel="icon">` tags in `index.html:21-24` and the marketing `BaseLayout.astro:67` with the per-size set.
- Effort: S

**[1.B4] manifest.json declares sizes that do not match the on-disk files.**
- File: `public/manifest.json:14-31`
- Evidence: lines 16, 23, 28 declare `192x192`, `180x180`, `1200x630` for files that are actually 2000×2000 each.
- Impact: Lighthouse PWA "icons match declared size" check fails. Chrome's installability heuristic may downgrade install prompt; on Android the home-screen icon is scaled from 2000→192 with no maskable safe area, producing a visibly cropped/blurry app icon. WebAPK on Android Chrome reads the 192 entry to generate a mini-APK.
- Fix: regenerate `public/icon-192.png` (192×192), `public/icon-512.png` (512×512), `public/icon-192-maskable.png` (192×192, with 20% safe-area padding, `purpose: "maskable"`), `public/icon-512-maskable.png`. Update `manifest.json` icons array to point at each. Match on-disk dimensions exactly.
- Effort: M

**[1.B5] No 512×512 PWA icon and no maskable icon.**
- File: `public/manifest.json:13-32`
- Evidence: only entries are `/favicon.png` (192), `/apple-touch-icon.png` (180), `/og-image.png` (1200×630). No `512x512`. No entry has `"purpose": "maskable"`.
- Impact: Chrome PWA install criteria require both 192 and 512 icons; without 512, WebAPK installation falls back to a generated icon. Without a maskable icon, Android adaptive launchers (Pixel, Samsung One UI) crop the icon's corners and risk cutting off the wordmark.
- Fix: add the 512 + maskable variants from [1.B4] to `manifest.json:13-32`.
- Effort: S (combined with B4)

### Critical

**[1.C1] Brand-aqua token is shipping the wrong color.**
- File: `src/index.css:27`
- Evidence: `--brand-aqua: 184 85% 44%; /* #11C4D0 - Primary Aqua */`. HSL(184, 85%, 44%) renders as **#11C4D0**, the comment confirms it. Brand spec is **#14C8CC**. Comparable: `--brand-aqua-dark: 186 85% 35%` → #0D99A8 (also off-spec, brand has no `#0D99A8` declared). The token file is the single source of truth used by all `bg-brand-aqua`, `text-brand-aqua`, etc. Tailwind classes (`tailwind.config.ts:42-49`).
- Impact: every aqua surface in the app — buttons, focus rings, links, primary CTAs, badges, the favicon-mask color — is shipping a slightly-off aqua. Audience won't catch the hex code, but a side-by-side compare with the brand guide will read "wrong." This is the canonical drift Jo's memory warns about.
- Fix: change `src/index.css:27` to `--brand-aqua: 182 82% 44%; /* #14C8CC - Primary Aqua */`. Update the inline comments on lines 27, 43 ("Primary - Aqua #11C4D0"). Recompute `--ring`, `--primary` to match (line 44, 77). Verify by adding a brand-color visual regression test that asserts `getComputedStyle().color` of `.text-brand-aqua` is `rgb(20, 200, 204)`.
- Effort: XS to fix; S to verify across all components.

**[1.C2] manifest theme_color and HTML mask-icon use #2D9A8C, which is neither brand color.**
- File: `public/manifest.json:8` (`"theme_color": "#2D9A8C"`); `index.html:26` (`<link rel="mask-icon" href="/favicon.png" color="#2D9A8C">`)
- Evidence: #2D9A8C is a desaturated teal. Brand aqua is #14C8CC; brand gold is #E4C875. #2D9A8C was likely lifted from the original design template and never swapped.
- Impact: Android Chrome paints the status bar and splash screen using `theme_color` — users will see a desaturated teal bar that does not match the in-app aqua. Splash screen on Android PWA install also samples this. iOS Safari ignores it. The mask-icon color also affects Safari pinned tab tint (when an SVG mask-icon exists; see [1.M1]).
- Fix: in `public/manifest.json:8` set `"theme_color": "#0F1112"` (carbon-black brand-dark, matches `index.html:30` `meta theme-color`) OR `"#14C8CC"` (brand aqua). Recommendation: brand-dark for both, since the app chrome reads dark. Update `index.html:26` mask-icon color to brand aqua `#14C8CC`. Same change in `marketing/src/layouts/BaseLayout.astro:70`.
- Effort: XS

**[1.C3] Template orange #F5B049 still present in v2-announcement.css.**
- File: `src/components/announcements/v2-announcement.css:11, 162, 163, 269, 300`
- Evidence: line 11 comment `*   --gold      → #F5B049`; line 162 `color: #F5B049;`; line 163 `-webkit-text-fill-color: #F5B049;`; line 269 `color: #F5B049;`; line 300 `color: #F5B049;`. Sibling files (`autopost-tokens.css:11-12`, `settings-tokens.css:11-12`, `support-tokens.css:11-12`) explicitly document that `#F5B049` was swapped to `#E4C875` — this file was missed.
- Impact: announcements/banners display orange, in direct violation of the brand rule. Per memory: orange in autopost/lab UI is forbidden. v2-announcement is shown on landing/dashboard.
- Fix: in `src/components/announcements/v2-announcement.css`, replace all `#F5B049` with `#E4C875` (5 occurrences); update the line 11 comment to `*   --gold → #E4C875`.
- Effort: XS

**[1.C4] mask-icon points to a PNG; Safari requires SVG.**
- File: `index.html:26` (`<link rel="mask-icon" href="/favicon.png" color="#2D9A8C">`)
- Evidence: the W3C/Apple pinned-tab `mask-icon` spec accepts only monochrome SVG. PNG is silently ignored by Safari.
- Impact: Safari pinned tabs render a default favicon (or the standard icon scaled), defeating the purpose of declaring the link.
- Fix: ship `public/favicon-mask.svg` (monochrome SVG of the M-mark, fill #000); change `index.html:26` to `<link rel="mask-icon" href="/favicon-mask.svg" color="#14C8CC">`. Same for `marketing/src/layouts/BaseLayout.astro` if the marketing site adds the link.
- Effort: S

### Major

**[1.M1] mmbg.svg is 1.8 MB.**
- File: `public/mmbg.svg` (1,889,015 bytes per `ls -la`)
- Evidence: SVGs at 1.8 MB almost always embed base64-encoded raster images, defeating vector compression. The hero background `herobackground.png` is 2.5 MB and `herobackground.webp` is 118 KB — vector should be far smaller than either.
- Impact: blocks first paint on mobile. Target audience is mobile-heavy; on a 4G connection (~5 Mbps), 1.8 MB adds ~3 s to LCP. Lighthouse "avoid enormous network payloads" fails.
- Fix: open `public/mmbg.svg` in an editor; strip embedded `<image xlink:href="data:image/...">` blocks or replace them with externally referenced WebP. Re-export from the source. Target <50 KB for a hand-authored SVG. If the asset is meant to be raster, rename to `mmbg.webp` and serve as such.
- Effort: M

**[1.M2] No PWA splash-screen images for iOS.**
- File: `index.html:21-31` (head links), `public/manifest.json`
- Evidence: no `<link rel="apple-touch-startup-image">` tags; no per-device-size splashes in `public/`.
- Impact: when a user adds MotionMax to iOS home screen, launching the PWA shows a white flash for ~1-2 seconds before React mounts. No brand presence at the most install-y moment.
- Fix: generate per-device splash PNGs (iPhone SE, 12, 14 Pro, 16 Pro Max, iPad mini, iPad Pro 11/12.9) at the iOS-required dimensions, save under `public/splash/`, add 14 `<link rel="apple-touch-startup-image" media="…" href="…">` tags to `index.html` head. Tools: `pwa-asset-generator` CLI.
- Effort: M

**[1.M3] msapplication-TileImage points at the 752 KB favicon.**
- File: `index.html:29` (`<meta name="msapplication-TileImage" content="/favicon.png">`)
- Evidence: Windows pinned-tile spec expects a 144×144 (small), 270×270 (medium), 558×270 (wide), 558×558 (large) image set referenced via `browserconfig.xml`. Pointing at a 2000×2000 favicon is the same wrong-asset bug.
- Impact: when a Windows user pins MotionMax via Edge "Pin to Start", the tile is generated from a 752 KB PNG. Either remove the meta or do it correctly.
- Fix: produce `public/mstile-150x150.png`, `public/mstile-310x150.png`, `public/mstile-310x310.png`; create `public/browserconfig.xml`; reference via `<meta name="msapplication-config" content="/browserconfig.xml">`. OR delete the line at `index.html:29` if Windows pinning is not a target surface.
- Effort: S (delete) or M (correct).

**[1.M4] No og:image variants per platform.**
- File: `index.html:36-49`, `marketing/src/layouts/BaseLayout.astro:56-65`
- Evidence: single `og:image` and `twitter:image` URL for all platforms. Twitter prefers 1200×675 (16:9), LinkedIn prefers 1200×627, Slack prefers 1200×630, WhatsApp prefers ~1.91:1, iMessage uses 1200×630 link previews.
- Impact: with one shared 2000×2000 asset (already broken — see [1.B1]), every platform crops differently. Even after [1.B1] is fixed with one 1200×630, Twitter previews will be off-aspect.
- Fix: produce two OG masters: `public/og-image.png` (1200×630, used by FB/LinkedIn/Slack/iMessage/Discord) and `public/og-image-twitter.png` (1200×675). Override `<meta property="twitter:image">` (`index.html:47`) to point at the Twitter variant.
- Effort: S

**[1.M5] theme-color mismatch between manifest and HTML head.**
- File: `public/manifest.json:8` (`#2D9A8C`); `index.html:30` (`<meta name="theme-color" content="#0F1112">`)
- Evidence: two different colors. Android Chrome chooses based on context (manifest for installed PWA, meta for browser tab).
- Impact: in-browser address bar and installed-PWA status bar display different brand colors. Inconsistent brand impression.
- Fix: align both to brand-dark `#0F1112` (recommended). Same fix as [1.C2].
- Effort: XS

**[1.M6] Marketing site has the same identical-PNG bug.**
- File: `marketing/dist/favicon.png`, `marketing/dist/apple-touch-icon.png`, `marketing/dist/og-image.png` (all SHA256 b125df082f… 2000×2000)
- Evidence: per `sha256sum` and `file` output above.
- Impact: marketing site (motionmax.io landing) shares broken share previews and bloated favicons. The marketing surface is the conversion entry point — broken OG cards cost click-through-rate.
- Fix: marketing build copies from `marketing/public/`. Replace those source assets in lockstep with the app fixes; verify `marketing/dist/` regenerates correctly via build.
- Effort: S (folded into [1.B*] fixes)

## §11 — Social Cards (Analytics/Marketing/Growth)

All §11 findings are downstream of the §1 OG image bug ([1.B1], [1.M4]). The relevant evidence for §11 specifically:

**[11.B1] Twitter/Facebook/LinkedIn/Slack share previews will all render broken at launch.**
- File: `index.html:36-49`, `marketing/src/layouts/BaseLayout.astro:56-65`, `public/og-image.png`
- Evidence: `og:image:width=1200`, `og:image:height=630` declared (`index.html:37-38`); actual image 2000×2000 (square). Twitter card type is `summary_large_image` (`index.html:43`) — square images are auto-cropped to a 1.91:1 horizontal slice from the center.
- Impact: every paid ad, every shared blog link, every signup link in transactional email, every social-share button in the app produces an embedded preview that crops or distorts the brand mark. Conversion metric: industry data puts the impact of a broken OG card at 30-50% lower CTR vs correct cards.
- Fix: same as [1.B1] — produce a real 1200×630 landscape composition; verify with `https://www.opengraph.xyz/url/` and Twitter's Card Validator before launch.
- Effort: S

**[11.C1] Single OG image used as `Organization.logo` in JSON-LD.**
- File: `index.html:107` (`"logo": "https://motionmax.io/og-image.png?v=20260129"`)
- Evidence: schema.org Organization expects `logo` to be a square or near-square image of the brand mark, not the share-card composition. Using og-image.png (which after fix will be 1200×630 landscape) misuses the field.
- Impact: Google Knowledge Panel and rich results may render the landscape OG card cropped to a small square thumbnail next to the org name — looks unprofessional in SERP.
- Fix: at `index.html:107` change `"logo"` to a square master e.g., `"https://motionmax.io/momaxlogo-512.png"`. Add a 512×512 brand-mark PNG to `public/`. Same change in marketing site if it duplicates the JSON-LD.
- Effort: XS

**[11.M1] No per-page OG override pattern.**
- File: `marketing/src/layouts/BaseLayout.astro:56-65` (uses single `ogImage` prop)
- Evidence: marketing layout accepts a single `ogImage` value. App `index.html` is fully static. No mechanism for per-blog-post or per-feature-page custom OG cards.
- Impact: every shared MotionMax URL, regardless of route, shows the same generic preview. Lower share-driven CTR vs personalized previews. As blog/SEO content grows (per `SEO_DISTRIBUTION_PLAN.md`), this becomes a bigger drag on growth.
- Fix: in `marketing/src/layouts/BaseLayout.astro`, accept `ogImage`, `ogTitle`, `ogDescription` props with sensible defaults; document in the layout. For the app, decide whether route-specific OG is needed (probably yes for `/share/<videoId>` deep links) and migrate to a server-rendered OG endpoint if so.
- Effort: M

**[11.M2] Twitter handle is set; no other social meta.**
- File: `index.html:49-50` (`twitter:site`, `twitter:creator`)
- Evidence: only Twitter is referenced. No `fb:app_id`, no `linkedin:owner`, no `pinterest:rich_pin`. Schema.org Organization `sameAs` only lists Twitter (`index.html:110-111`).
- Impact: missed analytics attribution on Facebook/LinkedIn/Pinterest shares; missed Instagram and TikTok presence (the audience lives on these platforms).
- Fix: once social accounts exist, add them to `sameAs` array at `index.html:110-111` and to the analogous block in marketing layout. Add `<meta property="fb:app_id">` when FB pixel is set up.
- Effort: XS to add when accounts exist.

## Summary
This category is one of the cheapest layers of work on the project (a single afternoon of asset generation handles 80% of the blockers) but until it lands, every external-facing surface — browser tab, iOS home screen, Android install, every social share, every transactional email link preview — broadcasts unfinished branding. Recommend gating the production launch on at minimum [1.B1] (OG), [1.B4] (manifest), [1.C1] (aqua token), and [1.C3] (orange remnant). The remaining work can ship within the first post-launch week without harming the launch credibility.

NEEDS: Pixel can produce the 1200×630 OG, the 192/512 maskable icons, and the corrected `--brand-aqua` token in a single subsequent task if dispatched. Effort: ~3 hours total.
DEADLINE: production gate.

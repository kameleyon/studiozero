## 2. Intended purpose + intended users

### 2.1 What Studio Zero does

Studio Zero performs automated **audits**, automated **builds**, and (at V2.1+) generates **working scaffold/MVP code**:

- **Audit mode (M1+).** Unchanged. Audits of software artifacts producing a graded checklist + readiness score + verdict.
- **Build mode — roadmap + docs + seeded repo (V2+).** Unchanged. Generative product planning from a customer-supplied idea description per `legal/ai-system-card-v1.5.md` §2.1. BigBrain brief gate (first human-oversight); multi-layer dispatch; Jury audit gate against the bundle (second structural gate); customer review before publication (second human-oversight gate). The bundle + seeded repo are the customer's property per `legal/build-mode-extension.md` §2.
- **Build mode — scaffold / MVP code (V2.1+ — LIVE at this card).** When the customer's `output_preference='roadmap_docs_seed_scaffold'` (PRD §7.3 step 5), Studio Zero **also** generates working scaffold/MVP code (Next.js + Tailwind + Supabase by default; templates per `project-template/`). The scaffold runs through a **separate Jury audit gate** (the scaffold-audit-gate Edge Function, distinct from the build-audit-gate that governs the bundle); the scaffold delivers only on PASS or PASS WITH FIXES. A FAIL refuses delivery — `scaffold_jobs.state` lands on `audit_failed` (DB-enforced via `scaffold_jobs_delivered_verdict_required` CHECK + the `scaffold_jobs_audit_gate_update` RLS predicate in `architecture/database/migrations/0010_scaffold_v2_1.sql`). The generated scaffold code is the customer's property per `legal/build-mode-extension.md` §2A.1 (V2.1 amendment). The customer reviews the scaffold before deploying — the V2.1 human-oversight gate (§8 below).

### 2.2 Intended users

### 2.3 Excluded use cases (explicit) — unchanged from v1.5; Scaffold mode does not relax

### 2.4 Out-of-scope as a deliverable

- **Not a security pentest.** Unchanged from v1.5. The audit gate is a quality screen, not a security test. **Scaffold mode adds:** the scaffold is **not** pentested before delivery — `legal/build-mode-extension.md` §2A.4 governs.
- **Not a legal opinion.** Unchanged. Audit-gate PASS on scaffold is not legal IP clearance, not a trademark check, not a patent search.
- **Not a substitute for human judgment.** Unchanged. Scaffold deliveries are advisory; customer decides whether to deploy. The §2A.5 customer-review obligation is the human-oversight gate before deploying.
- **NEW V2.1 — Not a production-readiness certification.** A PASS or PASS WITH FIXES verdict on the scaffold means the audit gate did not find disqualifying defects against the brief; it does **not** mean the scaffold is production-ready, bug-free, performant at any scale, or fit for any particular environment. `legal/build-mode-extension.md` §2A.4 documents this disclaim in plain language; the customer-review obligation in §2A.5 puts the production-readiness gate on the customer.

## 4. Training data summary

**Studio Zero does not train, fine-tune, or distill any model.** Unchanged from v1.0 + v1.5.

Studio Zero's own retention posture for V2.1 Scaffold mode:

- **Customer scaffold_jobs row (the row in `0010_scaffold_v2_1.sql` §C):** retained for the lifetime of the parent build per `legal/build-mode-extension.md` §2A + §8. Members may delete a build (member DELETE policy on `builds` in 0009 §H.1 cascades to `scaffold_jobs` via the FK CASCADE in 0010 §C). Tenant deletion cascades all scaffold data.
- **Generated scaffold artifact (build_artifacts row pointed to by `scaffold_jobs.generated_artifact_id`):** retained alongside the build. Immutable per the 0009 §E `build_artifacts` posture (members SELECT only; service-role INSERT only).
- **Signed URL audit witness (`scaffold_jobs.zip_download_url`):** the 24-hour signed URL TTL is enforced by Supabase Storage's signed-URL expiry; the column persists as a historical record of which URL was issued. The actual signed URL becomes useless after 24h; refresh is via Forge's re-issue endpoint.
- **No retraining loop.** Scaffold mode does not feed customer brief text, generated scaffold code, or audit findings to Anthropic for training. Anthropic's commercial API terms govern whether Anthropic retains prompts — we do not opt customers into any training arrangement.

## 6. Limitations

- **Hallucinations are possible.** Unchanged from v1.5. **Scaffold mode adds:** generated code may compile and pass review but still contain latent defects. The scaffold-audit-gate catches many defect classes (missing tests, contradictory configuration, accessibility regressions, brand-drift, undisclosed third-party dependencies); it does not catch all. Customer reviews before deploying (§8 + Build extension §2A.5).
- **Auto-PR patches require Jury re-audit gate before opening.** Unchanged.
- **Build mode bundles require Jury audit gate before delivery.** Unchanged.
- **NEW V2.1 — Scaffold-mode deliveries require the scaffold-audit-gate before delivery.** Verdict FAIL refuses delivery (`scaffold_jobs_delivered_verdict_required` CHECK in `architecture/database/migrations/0010_scaffold_v2_1.sql` §C + `scaffold_jobs_audit_gate_update` RLS in §D). Same structural-defense pattern as V2 bundles; the DB rejects unsafe transitions even if app code is buggy.
- **PRs are never auto-merged.** Unchanged.
- **Seeded repositories are not auto-modified after seeding.** Unchanged. **Scaffold mode adds:** the generated scaffold is delivered once (zip or seeded-repo Git history); Studio Zero does not maintain the scaffold after delivery. Subsequent fixes ship via Auto-PR if the customer separately enables it on the same repo.
- **No real-time monitoring of merged customer code.** Unchanged.
- **English-language primary support.** Unchanged. Scaffold-mode source code is in the language of the template (TypeScript/JavaScript for Next.js templates; YAML/JSON for configuration; etc.); README and inline comments are in English unless the brief explicitly requests otherwise.
- **Score-engine drift across rubric versions.** Unchanged.
- **Build/Scaffold mode is Managed Pro only at V2 / V2.1.** Unchanged. Customers on BYOK/CLI/Free/Managed Starter cannot use Scaffold mode without upgrading.
- **NEW V2.1 — Scaffold code may resemble publicly available references.** Same mechanism as v1.5 bundle limitation, now applied to executable code: the foundation model was trained on public source code, so generated scaffold code may bear stylistic, structural, or substantive similarity to public references. `legal/build-mode-extension.md` §4 + §2A.10 govern; customer verifies before publishing or deploying.
- **NEW V2.1 — Generated scaffold may declare third-party dependencies the customer did not explicitly request.** Scaffold templates ship with a baseline dependency set (Next.js, Tailwind, Supabase client, etc.) that the audit gate's dependency-license review evaluates. Customers are still expected to verify the final `package.json` (or equivalent manifest) before running `npm install` — the audit gate is a quality screen, not a license-compliance certification (§2A.4 + §2A.5 govern).
- **NEW V2.1 — Generated scaffold is not pentested before delivery.** The offline-network-tap test is a generation-time safety check (the scaffold generator does not POST customer code externally); it is not a delivered-code pentest. Customers deploying to production conduct their own security review.

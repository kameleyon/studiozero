## 2. Intended purpose + intended users

### 2.1 What Studio Zero does

Studio Zero performs automated **audits** AND, at V2+, automated **builds**:

- **Audit mode (M1+).** Audits of software artifacts — GitHub repositories, local folders, deployed URLs — producing a graded checklist of findings, a readiness score, a verdict (PASS / PASS WITH FIXES / FAIL), and, at V1.5+ for Code/Full SKU customers who purchase the Auto-PR upcharge, AI-authored code-change patches delivered via pull request against the customer's repository on a feature branch `studio-zero/fix-<run-id>`, gated by a Jury re-audit pass. **PRs are never merged automatically — customer reviews and merges.**
- **Build mode (V2+).** Generative product planning from a single customer-supplied idea description (PRD §7.3 Build Workflow). BigBrain produces a structured brief conforming to `architecture/schemas/brief.v1.schema.json`; the customer **reviews and confirms** the brief BEFORE dispatch (the first human-oversight gate). The multi-layer agent system (per `SYSTEM_ACHITECTURE.md` §3) executes its phases; layer-leads produce roadmap + PRD + architecture + brand tokens + voice document + decisions log + risks register + COGS analysis + channels analysis + README. The Jury audit gate runs against the generated artifact (PRD §7.3 step 7); the bundle delivers only on PASS or PASS WITH FIXES (the second human-oversight gate is the customer's review of the bundle before they ship anything built on it). At V2.1+, with the customer's `output_preference=roadmap_docs_seed_scaffold`, a working scaffold/MVP code is also produced — audit-gated identically. **The bundle and seeded repository are the customer's property** per `legal/build-mode-extension.md` §2.

### 2.2 Intended users

- Solo founders and indie agencies building software with AI builders (primary MVP audience per `PRD.md` §3a) — Audit mode + Build mode.
- Engineering teams seeking an independent audit before launch — Audit mode.
- **Non-technical solo founders** (Managed tier) seeking an idea → roadmap + docs bundle they can hand to a contractor — **Build mode primary persona** (PRD §5 row 1).
- Compliance teams measuring WCAG 2.2 AA conformance, FTC AI substantiation, brand-voice consistency — Audit mode.

### 2.3 Excluded use cases (explicit) — unchanged from v1.0; Build mode does not relax

Studio Zero is **not** to be used in the following contexts. The AUP at `legal/aup.md` enforces these prohibitions; service is suspended on detection. Build mode adds an **idea-side warranty** (`legal/build-mode-extension.md` §3) that you do not submit ideas for the following categories.

1. **High-risk Annex III contexts without additional controls.** Studio Zero may not be the sole or primary technical control in any context that Annex III classifies as high-risk. **Build mode adds:** you may not submit an idea description for a product in these contexts without a separate Comply use-case review.
2. **Safety-critical infrastructure** (kernel, medical device firmware, autonomous-vehicle, ICS/SIL, aviation DO-178C, nuclear). Same prohibition applies to Build mode idea submissions.
3. **Employment screening.** Build mode may not produce roadmaps for products whose purpose is employment screening of identifiable individuals.
4. **Lending or credit decisions.** Build mode may not produce roadmaps for ECOA/FCRA/Reg. B underwriting flows.
5. **Biometric identification or categorization.** Build mode may not produce roadmaps for biometric ID/categorization products.
6. **Surveillance of natural persons** under EU AI Act Art. 5 prohibited-practices. Build mode may not produce roadmaps for these.

Customers at the edge of the above should email `comply@studiozero.dev` for a use-case review before activating Build mode.

### 2.4 Out-of-scope as a deliverable

- **Not a security pentest.** Studio Zero identifies risk surface; it does not exploit vulnerabilities. Build mode scaffolds (V2.1+) are **not** pentested before delivery — the audit gate is a quality screen, not a security test (see `legal/build-mode-extension.md` §5).
- **Not a legal opinion.** A WCAG, compliance, or audit-gate finding is a technical observation, not a legal certification. **Build mode adds:** the audit gate's PASS on a bundle is not legal IP clearance, not a trademark check, not a patent search — see `legal/build-mode-extension.md` §5.
- **Not a substitute for human judgment.** The verdict, score, Auto-PR patches, and **Build mode bundles + seeded repos + scaffolds** are advisory; the customer decides whether to ship.

## 3. Underlying foundation models

Unchanged from v1.0. Studio Zero's AI capability is provided by **Anthropic's Claude** family of large language models accessed via Anthropic's commercial API. We do **not** retrain or fine-tune Claude models, and we do **not** operate our own foundation models. Studio Zero is a **downstream deployer** under the EU AI Act, not a general-purpose AI model provider.

### 3.1 Model pinning posture at v1.5

Studio Zero pins to specific Claude model versions per environment.

**Status at v1.5:** the `runner/llm/pinned-versions.json` artifact remains **HUMAN-pending** (Forge implementation carry-over from v1.0 §3.1). Runtime pin is still enforced at the LLM gateway via environment variables `ANTHROPIC_MODEL` and `ANTHROPIC_MODEL_FALLBACK` configured per-environment by Terra. The drift-dashboard nightly check operates against the gateway-config snapshot in the absence of the manifest file. Comply tracks Forge delivery to close before the next quarterly re-verification.

The V2 milestone graduates the runner to **Firecracker microVM** sandboxing (PRD §16 V2 + ARCH-D2 phased close) after a second clean external pentest (`sprint/milestone-V2.md` Shield deliverable). The model pin posture is orthogonal to the sandbox graduation; both are V2 deliverables but they close independently.

### 3.2 Authoritative model documentation

Anthropic's published model card per model is the authoritative source. Customers requiring a specific Claude version pin (e.g., a regulated industry that has accepted a particular Anthropic model card revision) request in writing to `comply@studiozero.dev`. Build mode customers using Managed Pro can rely on the same pin posture as audit mode — there is no separate model pinning for Build dispatches; layer-leads use the same pinned models.

## 4. Training data summary

**Studio Zero does not train, fine-tune, or distill any model.** Unchanged from v1.0.

Studio Zero's own retention posture for V2 Build mode:

- **Customer idea descriptions + brief + bundle artifacts + build_events log:** retained for the lifetime of the build row per `legal/privacy-policy.md` §3 + `legal/build-mode-extension.md` §8. Members may delete a build (member DELETE policy on `builds` in `architecture/database/migrations/0009_build_mode_v2.sql` §H.1). Tenant deletion cascades reap all build data.

## 6. Limitations

- **Hallucinations are possible.** Unchanged from v1.0. Build mode amplifies this risk by length: a bundle is a multi-document artifact, and Jury audit gate is the structural defense (the gate catches inconsistency across documents and falsifies fabricated decisions). Customers are still expected to review the bundle before publishing or building against it (`legal/build-mode-extension.md` §10).
- **Auto-PR patches require Jury re-audit gate before opening.** Unchanged.
- **Build mode bundles require Jury audit gate before delivery.** Verdict FAIL refuses delivery (`builds_delivered_verdict_required` CHECK in `architecture/database/migrations/0009_build_mode_v2.sql` §D). This is the load-bearing structural defense for Build mode.
- **PRs are never auto-merged.** Unchanged.
- **Seeded repositories are not auto-modified after seeding.** Studio Zero's GitHub App's seeding permission terminates with the seeding event unless the customer separately enables Auto-PR on the same repo. The bundle's roadmap and scaffold are starting points, not maintained code.
- **No real-time monitoring of merged customer code.** Unchanged.
- **English-language primary support.** Unchanged. Build mode's PRD, README, and voice document are emitted in English unless the brief explicitly requests otherwise.
- **Score-engine drift across rubric versions.** Unchanged. Build mode bundle's `score_engine_version` field is the same versioning surface as audit mode's verdict.
- **Build mode is Managed Pro only at V2.** Customers on BYOK, CLI, Free, or Managed Starter cannot use Build mode without upgrading. The base Audit mode flow continues without change.
- **Build mode bundles may resemble publicly available references.** Foundation-model training data includes public material; bundle output may bear stylistic, structural, or substantive similarity to public references (`legal/build-mode-extension.md` §4). Customers verify before publishing.

## 8. Human oversight + transparency commitments

Beyond what EU AI Act Article 50 requires:

- **Disclose on every API response.** Unchanged from v1.0. `X-AI-Generated: studio-zero` HTTP header + `<meta name="ai-generated" content="studio-zero">` on every emitted HTML page.
- **Disclose in every Auto-PR.** Unchanged. Each PR opens with the Art. 50 disclosure paragraph from `legal/pr-body-template.md`; each commit carries `AI-Authored: studio-zero/runner@v<x.y.z>`.
- **Disclose in every Build mode bundle.** **NEW at V2.** Every artifact in a delivered bundle — the README, the PRD, the architecture doc, the brand tokens file, the voice document, the decisions log, the risks register, the COGS analysis, the channels analysis — opens with a short disclosure stating: "This document was generated by Studio Zero v1.5 from a customer-confirmed brief. AI-generated content (Art. 50 EU AI Act 2024/1689). Reviewed by the Studio Zero audit gate before delivery; customer reviews before publication." The text of this disclosure is locked verbatim by Comply, the same way the PR-body Art. 50 paragraph is. The seeded repository's README leads with this disclosure paragraph; the V2.1+ scaffold's source-tree top-level NOTICE file carries it.
- **Customer reviews the BigBrain brief BEFORE dispatch.** **NEW at V2.** The first human-oversight gate for Build mode. State machine pauses at `brief_pending_confirmation` (migration 0009 §A); customer ticks the confirmation CTA; `audit_action='build_brief_confirmed'` ledger row is written; dispatch starts. The customer may edit the brief at this stage (members have UPDATE on builds per migration 0009 §H.1) without burning Managed Pro inclusion (extension §6.1).
- **Customer reviews the bundle BEFORE delivery.** **NEW at V2.** The Jury audit gate is the structural defense, but the customer is the human oversight: the delivered bundle is reviewed before the customer publishes, deploys, or builds against it. The bundle is "advisory" in the same sense an audit verdict is advisory (`legal/build-mode-extension.md` §10).
- **Jury audit gate.** **NEW at V2.** Same Edge-Function-only RLS predicate as ARCH-D7 (Auto-PR) — only the `build-audit-gate` Edge Function (JWT iss + role) may transition `builds.state` to `audit_gate_passed` or `audit_gate_failed` (migration 0009 §H.1). The DB rejects the transition if app code is buggy or the gate is bypassed. Paired with the `builds_delivered_verdict_required` CHECK: FAIL verdict can NEVER reach delivery.
- **Customer review before merge (Auto-PR).** Unchanged from v1.0.
- **Opt-out of Auto-PR.** Unchanged.
- **Opt-out of Build mode.** Don't subscribe to Managed Pro at V2; the audit-only flow continues. Mid-flow cancellation (before `layers_dispatching`) is refundable per `legal/build-mode-extension.md` §6.4 + `compliance/d22-cooling-off-flow.md`.
- **Publish this System Card.** LIVE at `https://studiozero.dev/system-card`; updated quarterly minimum or on material change.
